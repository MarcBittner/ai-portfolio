# Deployment (optional): Docker + GCP Cloud Run + Redis

The service is a single container: FastAPI + the built frontend served
from one origin. Offline mode works in the container too — with no env
vars it runs the mock/in-memory stack, which makes `docker run` a
zero-config smoke test.

## Build & run locally

```sh
cd projects/persona-twin
docker build -t persona-twin .
docker run --rm -p 8080:8080 persona-twin
# UI at http://localhost:8080/  ·  API at /health, /api/ask, ...
```

Real backends are env vars, same as everywhere else:

```sh
docker run --rm -p 8080:8080 \
  -e ANTHROPIC_API_KEY=... -e MONGODB_URI=... -e REDIS_URL=... \
  persona-twin
```

> The image installs the base (offline) dependencies. To bake in
> provider extras: `RUN pip install --no-cache-dir ".[anthropic,openai,mongo,redis]"`.

## Local Kubernetes (kind)

Tested against a local [kind](https://kind.sigs.k8s.io/) cluster:

```sh
cd projects/persona-twin
docker build -t persona-twin:v0.2.0 .
kind load docker-image persona-twin:v0.2.0 --name YOUR_CLUSTER
# (equivalent without the kind CLI:
#  docker save persona-twin:v0.2.0 | docker exec -i YOUR_CLUSTER-control-plane \
#    ctr --namespace=k8s.io images import -)
kubectl apply -f deploy/k8s/persona-twin.yaml
kubectl -n persona-twin rollout status deployment/persona-twin
kubectl -n persona-twin port-forward svc/persona-twin 8080:80
# UI at http://localhost:8080/ · API at /health, /api/ask
```

The manifest (`deploy/k8s/persona-twin.yaml`) runs non-root with
readiness/liveness probes on `/health`, a NodePort (30080) for
node-level access, and `imagePullPolicy: Never` since the image is
side-loaded. Real backends go in a Secret (`envFrom` block is stubbed
in the manifest) — never literal env values.

## Cloud Run

```sh
PROJECT=YOUR_PROJECT_ID REGION=us-central1

# one-time: Artifact Registry repo
gcloud artifacts repositories create persona-twin \
  --repository-format=docker --location=$REGION

# build + push (Cloud Build)
gcloud builds submit --tag \
  $REGION-docker.pkg.dev/$PROJECT/persona-twin/api:latest

# deploy — either flag-style:
gcloud run deploy persona-twin \
  --image $REGION-docker.pkg.dev/$PROJECT/persona-twin/api:latest \
  --region $REGION --allow-unauthenticated --memory 512Mi

# or from the committed service YAML (edit placeholders first):
gcloud run services replace deploy/cloudrun-service.yaml --region $REGION
```

**Secrets:** never as literal env values. Store in Secret Manager and
reference them (`--set-secrets ANTHROPIC_API_KEY=anthropic-api-key:latest`,
or the `secretKeyRef` blocks in the YAML).

## Redis caching

`REDIS_URL` switches the cache port from the in-process LRU to Redis.
What's cached, with hit/miss counters visible in `/health`:

| Kind | Key includes | Why |
|---|---|---|
| `query_embedding` | embedder type + dims + question text | same/similar questions recur; embedding is the per-request constant cost |
| `answer` | persona + k + question + active llm backends | full `/ask` result for non-debug requests (TTL 1h); debug requests always recompute and show the cache state instead |

The backend set is part of the answer key on purpose: flipping a
provider on/off must not serve answers generated by a different stack.

On Cloud Run, use **Memorystore for Redis**: create the instance, add a
**Serverless VPC Access connector**, uncomment the connector annotation
in the YAML, and point `REDIS_URL` at the instance's private IP. Note
that with `minScale: 0` the in-process cache is cold on every cold
start — that's the actual argument for Redis here, not raw speed.

## What this setup deliberately omits

Auth on the API, TLS termination (Cloud Run provides it), multi-region,
and CI/CD wiring — out of scope for a reference implementation; the
container is the part with reusable decisions in it.
